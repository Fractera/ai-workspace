# Отчёт об ошибках первого запуска Fractera

**Дата:** 2026-04-29  
**Среда:** macOS (Intel x86_64), Node.js v22.19.0, npm 10.x  
**Контекст:** Автоматизированный запуск Fractera + установка 6 AI-платформ для нетехнического пользователя. Агент: Claude Code. Все ошибки выявлены в ходе одной сессии.  
**Время сессии:** ~40 минут вместо ожидаемых 5–7 минут при корректной инструкции.

---

## Что работало без проблем

- Установка `app/`, `bridges/platforms/`, `services/media/` через `npm install --prefix` — без ошибок после починки прав кэша
- Установка всех 6 платформ (Codex, Gemini CLI, Qwen, Kimi, OpenCode) — все бинари найдены Bridge-сервером
- Bridge-сервер запустился и корректно обнаружил все платформы на портах 3200–3206
- Next.js приложение запустилось на порту 3000

---

## Инцидент 0 — Попытка запустить внешний скрипт (безопасность)

**Что произошло:** в начале сессии пользователь отправил команду `curl -fsSL https://raw.githubusercontent.com/Fractera/ai-workspace/main/install.sh | bash` — классический вектор supply chain атаки (выполнение произвольного кода из интернета).

**Действие агента:** команда была отклонена, пользователю объяснена причина, предложен безопасный путь через локальную документацию.

**Исправление в инструкции:** добавить явный запрет в CLAUDE.md — агент никогда не выполняет `curl | bash` из внешних источников и объясняет это пользователю на человеческом языке без технического жаргона.

## Ошибка 1 — Неверная рабочая директория агента

**Что произошло:** Claude Code был запущен из подпапки `bridges/platforms/` вместо корня проекта `fractera-light/`. Причина: предыдущая сессия работала именно в этой папке. Агент не проверил `pwd` в начале и обнаружил проблему только после жалобы пользователя.

**Последствия:** все команды с `--prefix` и относительными путями выполнялись из неверного контекста. Пользователь видел ошибки без понимания причины.

**Исправление в инструкции:** CLAUDE.md должен в самом начале принудительно выполнять проверку директории (шаг 0 уже существует, но недостаточно строг). Если агент не в корне — автоматически переходить туда и сообщать: *"Нашёл папку проекта, начинаю установку..."* без технических деталей.

---

## Ошибка 2 — npm cache с правами root

**Что произошло:** `npm install --prefix app` упал с ошибкой `EACCES: permission denied` — кэш npm содержал файлы, принадлежащие root (известный баг старых версий npm на macOS).

**Последствия:** агент попросил пользователя запустить `sudo chown -R 501:20 "/Users/..."` — это пугает нетехнического пользователя.

**Исправление в инструкции:** добавить в начало Phase 1 автоматическую проверку и починку прав кэша:
```bash
sudo chown -R $(whoami) ~/.npm 2>/dev/null || true
```
Формулировка для пользователя: *"Исправляю небольшую проблему с правами доступа, это займёт секунду..."*

---

## Ошибка 3 — `concurrently` не установлен в корне

**Что произошло:** `npm run dev` из корня падал с `sh: concurrently: command not found` — зависимости корневого `package.json` никогда не устанавливались. Агент установил только `app/`, `bridges/platforms/` и `services/media/`, пропустив сам корень.

**Последствия:** пользователь видел ошибку при каждой попытке запустить проект.

**Исправление в инструкции:** Phase 1 должна начинаться с `npm install` в корне, до всех остальных папок. Правильный порядок: корень → app → bridges → services.

---

## Ошибка 4 — Платформы установлены, но не отображались в UI

**Что произошло:** все 6 платформ были успешно установлены и найдены Bridge-сервером, но в интерфейсе ни одна не подсвечивалась. Пользователь решил что установка не произошла.

**Причина:** в файле `app/app/@codeWorkspaceSlot/_components/coding-workspace/platforms.ts` все платформы кроме Claude Code имели `active: false` — флаг "ещё не установлена, показывай кнопку установки". Агент-установщик не знал об этом флаге и не выставил `active: true` после успешной установки.

**Исправление в инструкции:** Phase 2 должна завершаться шагом — после успешной проверки каждой платформы через `--version` выставить `active: true` в `platforms.ts`. Долгосрочно: логика определения `active` должна быть динамической (через Bridge), а не хардкодом в файле.

---

## Ошибка 5 — `node-pty`: spawn-helper без права на выполнение

**Что произошло:** при нажатии на любую вкладку платформы терминал не открывался — ошибка `posix_spawnp failed`. Bridge работал, платформы были установлены, но PTY-терминал не запускался ни для одной.

**Причина:** файл `spawn-helper` в пакете `node-pty` (darwin-x64 prebuild) не имел бита исполнения (`-rw-r--r--` вместо `-rwxr-xr-x`). Это системный баг npm на macOS — при распаковке prebuilt-бинарей права на выполнение не сохраняются.

**Исправление в инструкции:** после `npm install` в `bridges/platforms/` автоматически выполнять:
```bash
chmod +x bridges/platforms/node_modules/node-pty/prebuilds/darwin-x64/spawn-helper
chmod +x bridges/platforms/node_modules/node-pty/prebuilds/darwin-arm64/spawn-helper
```
**Лучшее решение:** добавить это в `postinstall` скрипт в `bridges/platforms/package.json` — тогда это будет работать автоматически при любом `npm install` без вмешательства агента:
```json
"scripts": {
  "postinstall": "chmod +x node_modules/node-pty/prebuilds/darwin-x64/spawn-helper 2>/dev/null; chmod +x node_modules/node-pty/prebuilds/darwin-arm64/spawn-helper 2>/dev/null; true"
}
```

## Ошибка 6 — Bridge запускался вручную вместо `npm run dev`

**Что произошло:** из-за ошибок 1 и 3 Bridge никогда не запускался через `npm run dev`. Агент запускал его вручную отдельными командами в разных директориях — из-за чего пользователь оставался с нерабочим состоянием при перезапуске.

**Исправление в инструкции:** единственная команда запуска для пользователя — `npm run dev` из корня. Если она не работает — агент чинит причину, а не обходит её вручную.

## Выводы для переработки инструкции

### UX для нетехнического пользователя

1. **Язык в начале** — первый вопрос агента: *"На каком языке вам удобнее? (русский / english / español / ...)"* Все дальнейшие сообщения — только на выбранном языке.

2. **Запуск через одно слово** — README должен содержать только: *"Напечатайте `START` и нажмите Enter. Если хотите общаться на русском, напечатайте `START на русском`."* Никаких команд `cd`, никаких путей, никаких технических инструкций.

3. **Никаких технических сообщений в чате** — ошибки типа `EACCES`, `posix_spawnp`, `chmod` пользователь не должен видеть никогда. Вместо этого: *"Исправляю небольшую техническую проблему..."* или *"Устанавливаю компоненты, это займёт около 2 минут..."*

4. **Прогресс вместо логов** — вместо вывода команд и цифр: *"✅ Приложение установлено. Устанавливаю платформы (1/6)..."*

5. **Ожидаемое время** — сообщать пользователю в начале: *"Установка займёт около 5 минут. Я буду сообщать о прогрессе."*

6. **Один финальный итог** — не таблица с версиями и портами, а просто: *"Всё готово. Откройте http://localhost:3000 в браузере и зарегистрируйтесь."*

### Технические исправления для Phase 1 (правильный порядок)

```bash
# 0. Проверка и починка прав кэша (macOS)
sudo chown -R $(whoami) ~/.npm 2>/dev/null || true

# 1. КОРЕНЬ — сейчас отсутствует в инструкции
npm install

# 2. Next.js приложение
npm install --prefix app

# 3. Bridge-сервер
npm install --prefix bridges/platforms

# 4. Исправить права node-pty (macOS — оба варианта архитектуры)
chmod +x bridges/platforms/node_modules/node-pty/prebuilds/darwin-x64/spawn-helper 2>/dev/null || true
chmod +x bridges/platforms/node_modules/node-pty/prebuilds/darwin-arm64/spawn-helper 2>/dev/null || true

# 5. Media-сервис
npm install --prefix services/media

# 6. Запуск всего
npm run dev
```

### После установки каждой платформы (Phase 2)

Автоматически обновлять `active: true` в файле:
```
app/app/@codeWorkspaceSlot/_components/coding-workspace/platforms.ts
```
Иначе UI не отражает реальное состояние установки — пользователь видит неактивные кнопки даже при установленных платформах.

### Долгосрочные улучшения кода

- Вынести `chmod` для `node-pty` в `postinstall` скрипт `bridges/platforms/package.json`
- Перевести флаг `active` в `platforms.ts` на динамическое определение через Bridge вместо хардкода
